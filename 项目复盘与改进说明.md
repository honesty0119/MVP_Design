# 项目需求复盘与改进说明

## 对题目的重新理解

这不是一个通用 CRM，也不是单纯的数据列表。题目要验证的是：候选人能否把“市场获客—外呼清洗—加微承接—销售跟进—MQL—SQL”拆成可执行、可约束、可追踪的业务闭环，并在 6–8 小时内做出可运行、可解释的 MVP。

因此，系统的核心不是页面数量，而是以下四件事：

1. 状态有业务含义，不能任意跳转。
2. 每次状态变化与跟进都能追溯。
3. MQL/SQL、无效、超时等口径能解释。
4. 外部系统重复、延迟、乱序回调不会轻易破坏当前状态。

## 截图与初版实现暴露的问题

- 页面完成度尚可，但主要信息仍是列表，漏斗阶段关系不够直观。
- 转化率只有数字，没有展示分子/分母，评审者容易追问指标为何这样算。
- “超 48h 未跟进”只能看数字和红字，不能直接筛出待处理线索。
- CSV 导入和去重只有 API，无法在现场通过页面完整演示。
- 时间以 ISO 字符串显示，偏开发调试界面，不利于业务人员快速读取。
- 三方外呼回调只做事件落库，没有展示“合法推进状态、重复事件不重复执行、旧事件不回滚后续状态”的闭环。
- 详情页没有在操作位置提示 MQL/SQL 准入规则，错误只能在提交后发现。
- 只有“能不能转状态”的工程提示，缺少业务评分解释：为什么这条线索值得培育、为什么还不能转 MQL。
- Agent 区域能回答问题，但如果不展示工具调用过程，现场容易被理解成普通聊天包装。

## 本轮改进

- 增加横向状态漏斗，并保留每个阶段的当前数量。
- 为有效率、MQL 转化率、SQL 转化率补充分子/分母说明。
- 超时指标支持一键过滤，补充重置筛选和搜索框回车触发。
- 增加 CSV 导入页面，明确表头、500 条上限、重复线索不覆盖原归因。
- 将时间格式改为适合业务阅读的月日时分。
- 外呼回调按 `event_id` 幂等；只有待外呼状态可被推进到已接通/未接通，旧回调不会把后续阶段拉回。
- 详情页直接展示 MQL/SQL 准入条件。
- 详情页增加线索评分和 MQL 判定解释，拆出联系方式有效性、负责人、跟进事实、渠道意向、私域承接和跟进时效。
- 页面顶部增加当前操作人，服务端校验销售/主管边界；销售只能操作自己负责的线索，主管可做负责人转移和无效线索复核。
- 新增审计记录，关键写操作记录操作人、动作和时间。
- Agent 回答区展示执行链路：收到问题、工具选择、参数、服务端校验、结果回填和数据边界。
- 新增外呼回调幂等测试，连同手机号去重、禁止跨级、无效原因必填共同形成关键边界测试。
- 新增评分解释、销售权限和 Agent 执行链路测试，避免新增能力只停留在页面展示。

## 仍然明确不做的范围

- 不把 Demo 伪装成生产系统：当前只有角色模拟，没有真实登录、完整 RBAC、消息队列和真实第三方连接。
- 不把手机号唯一当成长期客户主键：生产环境应区分客户主体、线索记录和归因窗口。
- 不把当前存量漏斗当成正式经营分析：生产指标应按进入时间 cohort 计算。
- 不用前端隐藏按钮代替权限控制：当前权限已在服务端校验，生产还需接入身份、团队数据范围和审批链路。
