# 3–5 分钟演示脚本

## 0:00–0:30：目标与边界

“这是一个市场线索从进入、外呼清洗、销售承接到 MQL/SQL 的简化闭环。重点不是做完整 CRM，而是验证状态规则、过程留痕、指标口径和异常处理。所有数据都是模拟数据。”

## 0:30–1:10：漏斗与业务问答

1. 展示顶部漏斗和每个阶段数量。
2. 指出指标卡下方的分子/分母口径。
3. 查询“本周哪个渠道 SQL 转化率最高？”。
4. 展示“LLM Agent”执行模式、`channel_sql_conversion` 工具名、参数和 Agent 执行链路。
5. 说明模型只负责选工具，服务端执行白名单参数化查询；没有 API Key 时会明确显示“规则回退”。

## 1:10–1:40：超时与筛选

1. 点击“超 48h 未跟进”，只看待处理线索。
2. 说明超时不会自动修改业务状态，它是运营提醒。
3. 点击重置，演示状态、来源和负责人筛选。

## 1:40–2:20：新增、导入与去重

1. 新增一条 11 位模拟手机号线索。
2. 再次新增同一号码，说明系统返回原线索，不覆盖原归因。
3. 打开 CSV 导入，说明表头、500 条限制和逐行错误统计。

## 2:20–3:40：负责人、评分、跟进与状态机

1. 打开线索详情，展示线索评分、加分项和 MQL 阻塞项。
2. 分配负责人并增加跟进记录，观察评分和“是否可转 MQL”的变化。
3. 说明 MQL 硬性准入是负责人 + 跟进，评分是业务解释层，不替代状态机。
4. 演示按合法路径变更状态。
5. 尝试跨级或不填写无效原因，展示服务端拒绝。
6. 展示状态历史和审计记录，强调跟进、状态和操作人只追加、不覆盖。

## 3:40–4:10：权限、三方回调与工程边界

“页面顶部可以切换当前操作人。销售只能操作自己负责的线索，主管可以转移负责人和复核无效线索。外呼回调使用 event_id 去重；只有待外呼状态能被推进到已接通/未接通。重复回调不重复执行，延迟回调不会把 MQL/SQL 拉回旧状态。生产环境还需要验签、重试、死信和乱序版本。”

## 4:10–4:40：AI-coding 与下一步

展示 `AI-coding使用说明.md` 中两个 Prompt：

- AI 最初允许状态任意跳转，我将规则收口到服务端状态机。
- AI 最初按手机号直接处理回调，我补充 event_id 幂等和防回滚。

最后说明下一步优先级：真实登录/RBAC、回调可靠性、cohort 漏斗与渠道成本归因。

## 备用追问回答

- 如果每天出现大量无效线索：先按来源、渠道、落地页版本和无效原因聚合，不能只看线索总量。
- 如果手机号多渠道重复进线：Demo 保留首次归因；生产要拆客户主体和线索记录，再按归因窗口处理。
- 如果 Agent 被问到超出范围的问题：不执行任意 SQL，只返回可解释的规则回退或建议问题。
