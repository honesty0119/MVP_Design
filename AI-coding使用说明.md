# AI-coding 使用说明

## 使用工具与参与环节

本作业使用 Codex 辅助完成：需求拆解、状态机与表结构草拟、API/页面代码生成、边界测试、README 和业务文档排版。我的职责是确定业务口径、审查生成结果、收缩范围并验证可运行性。

## AI 输出中发现的问题

初始生成思路容易把状态当成可任意修改的枚举，忽略“有效线索不能直接跳 SQL”、MQL/SQL 准入条件、无效原因必填和历史追溯；指标也容易只算状态数量，不说明分母与周期。此外，第三方回调如果只按手机号更新，会在重复投递时产生重复副作用。

## 两个真实 Prompt 与人工修正

### Prompt 1：状态流转

> 请为市场线索系统设计状态流转，覆盖新线索、外呼、有效/无效、加微、MQL 和 SQL，并生成状态修改接口。

AI 初稿把 `status` 设计成任意字符串更新，前端可以直接从新线索选择 SQL。这个输出“功能上能改状态”，但会破坏漏斗和阶段耗时，因此没有直接采用。我将允许路径收口到服务端 `ALLOWED_TRANSITIONS`，并补充三条准入规则：无效原因必填；MQL 需要负责人和跟进；SQL 只能由 MQL 转化且需要商机说明。对应测试验证跨级跳转会被拒绝。

### Prompt 2：第三方外呼回调

> 模拟外呼系统回调，根据手机号和接通结果修改线索状态。

AI 初稿建议收到回调后直接按手机号更新状态。这里有两个问题：第三方通常至少投递一次，同一事件可能重复；延迟回调还可能把已经进入 MQL 的线索拉回“已接通”。我改为要求 `event_id`，使用唯一约束实现幂等；只有当前状态为“待外呼”时才允许推进到“已接通/未接通”，其他状态只记录事件、不回滚业务状态。

## 我的判断与修正

- 将状态更新收口到服务端状态机，不由前端决定合法性。
- MQL 要求已分配负责人且至少有一次跟进；SQL 必须从 MQL 转化并填写商机说明。
- 无效必须填写原因；重新打开要求备注，并明确生产环境还需主管权限。
- 状态变化写入独立历史表，跟进记录不覆盖，便于审计和计算阶段耗时。
- 回调以三方 `event_id` 唯一约束实现幂等；手机号唯一约束用于线索去重，两者不混用。
- 明确漏斗为“当前存量快照”口径，不把 Demo 指标冒充正式经营口径。
- 使用 Python 标准库实现，避免评审环境因依赖安装失败而无法启动。

## 验证

实际启动服务并通过浏览器检查主流程；自动化测试覆盖手机号去重、禁止跨级跳转、无效原因必填、回调幂等、完整的新线索到 SQL 流程，以及自然语言指标查询口径。所有演示数据均为虚构号码。
